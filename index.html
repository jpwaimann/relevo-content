Search volume integration (Ahrefs / SEMrush)

  • Today: topics are generated blind from the rulebook plus WebSearch coverage. No keyword volume signal informs topic selection.
  • v2: pull Ahrefs or SEMrush data per topic candidate so the LLM can rank or filter ideation against real search demand.
  • Blocker for MVP: no API access is budgeted yet; bringing it in would also slow ideation while we tune the prompt to weigh volume sensibly against editorial judgment.

Automatic priority logic (P0 / P1 / P2)

  • Today: every topic written to the inventory is hard-coded to priority = P1. No per-topic prioritization decisions are being made.
  • v2: priority is assigned automatically using a combination of:
    • Search volume (via Ahrefs / SEMrush — depends on the integration above)
    • Sports calendar (football pieces lifted to P0 during May–June 2026 because of the FIFA World Cup; tennis to P0 during Grand Slam weeks; F1 to P0 over race weekends)
    • Coverage gap analysis (sports with fewer existing articles get a P0 lift to balance the catalog)
  • Blocker for MVP: the human-in-the-loop process for priority review is not designed yet. We are deferring until launch is behind us and we have observed what reviewers actually want to see in the queue.

Sports data API feed for Origins ideation

  • Today: Origins topics use WebSearch to surface relevant current players. This carries risk: stale information, missed breakout athletes, inaccurate ages or teams.
  • v2: integrate a sports data API (e.g. SportRadar, API-Football) so Origins ideation has fresh per-sport context — top scorers, recent transfers, injury comebacks, breakout rookies. Eventually this could auto-suggest player candidates rather than relying on the editor's mental model.
  • Blocker for MVP: cost and integration time. WebSearch combined with editorial curation is good enough for the first production batches.

Project overview

Relevo Content is an automated SEO content generation system for relevo.com. It produces editorial-quality Spanish-language articles across 24 sports and three content categories (Rules & Basics, Tactics Explainers, Origins) using a multi-pass pipeline that includes automated critique and fact-checking before any human review.

The system is implemented as a custom skill running inside Claude Code, replacing an earlier n8n-based workflow that was retired on May 14.

Why we rewrote it

The n8n implementation worked for a limited topic set but had three structural limitations that made it expensive to evolve:

  1. Prompt rigidity. Editorial voice was encoded inside individual workflow nodes. Any change to the style guide or section structure required a coordinated PR and redeploy across multiple nodes.
  2. No quality gate. The pipeline produced a draft and routed it directly to a human reviewer with no automated critique in between. Errors that an automated checker would have caught — factual inventions, length misses, format inconsistencies — reached editors.
  3. Limited coverage. Only 15 sports and two content categories were modeled. Expansion required editing multiple workflow branches by hand.

The Claude Code skill addresses all three. Prompts and templates are versioned markdown files that anyone on the content team can read and propose changes to via pull request. Every draft passes through automated critique against a 15-point editorial checklist and a fact-check stage before reaching a human reviewer. Coverage expands to 24 sports and introduces Origins (athlete biographies) as a third content category.

Pipeline architecture

Each article is produced by a deterministic seven-stage pipeline:

  1. Research — the skill reads the official rulebook PDF for the sport (stored in SharePoint) and supplementary authoritative sources.
  2. Draft — first version generated against the category template (Rules / Tactics / Origins) and the Relevo style guide.
  3. Critique pass 1 — an internal critic evaluates the draft against the 15-point checklist and produces a 0–100 score plus a list of priority fixes.
  4. Revise — if score is below 80 or anti-hallucination check fails, fixes are applied. If the draft passes the gate, this stage is skipped.
  5. Critique pass 2 — the revised version is re-scored. The loop repeats until the gate passes or retries are exhausted.
  6. Fact-check — claims are verified; unverifiable specific attributions are softened or removed.
  7. Publish — the article is uploaded to WordPress as pending_review and a notification subtask is created in Asana for the assigned reviewer.

The full intermediate state of every article — the pre-revise draft, both critique JSONs, the final HTML, the metadata bundle — is preserved on disk, giving editors complete traceability into how each piece was produced.

What reviewers are asked to evaluate

  1. Voice and tone. Do these read like Relevo articles? Are there expressions or rhythms in the style guide that should change?
  2. Category templates. Do Rules, Tactics, and Origins each have the structure that the team expects? Are there missing sections or sections that should be removed?
  3. Anti-hallucination behavior. The pipeline aggressively removes unverifiable specific attributions. Is that calibrated correctly, or too conservative for the kind of writing Relevo wants?
  4. Asana review workflow. Round-robin assignment, SLA, criteria for return-vs-approve — open for definition by the editorial team.
