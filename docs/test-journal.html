Scaling implications

The cycle is dominated by the critique–revise loop (~60% of wall-clock) and image generation (~15%). For the Phase 5 target of 50 articles per day:

  • Serial (one run at a time): 50 × 13 min = ~10.8 hours of wall-clock. Not viable for a single editor session.
  • Parallel (multiple runs concurrently, like the May 15 batch): the May 15 batch achieved ~1.7 min effective per article running 10 in parallel without image-gen. Image generation partially serialises through LiteLLM's rate limits, so the realistic figure for the full pipeline is 2–3 min effective per article in parallel. That puts 50 articles/day at ~2 hours of wall-clock with one editor dispatching the batch.
  • The actual bottleneck at production scale is not generation — it is the human review SLA. As long as reviewers (Joel and future round-robin pool members) validate at the pace the pipeline produces, the system sustains 50/day without queueing.

Variability and how to refine these numbers

The autonomous E2E run is a single sample. Duration will vary with:

  • Size of the rulebook (Cycling UCI is larger than IFAB; that lengthens the research read).
  • Number of revise passes required (this article needed 1; some will pass critique 1 on the first try and skip revise; pathological cases will need 2).
  • Latency of LiteLLM iGaming at the moment of the image call.

To get a stable baseline, the recommendation is to run 5–10 more articles end-to-end before the May 22 milestone and update the table above with the mean and the spread. ~12–15 minutes per article single-track is the working number for capacity planning until that data lands.

What got caught by running the real pipeline

Two production-relevant issues that earlier piecewise tests had not surfaced:

  1. Helper invocation signature drift. The sharepoint_msal.update_table_row() helper wraps the values argument internally; older call sites in scratch scripts wrapped it again, producing a Graph 400. The skill's lib/inventory.md shows the correct unwrapped form, so anything generated via the skill will be fine — but any helper code written by hand needs to know.
  2. Schema column drift between docs and live. The skill docs (and earlier conventions) had drifted on the order of columns 17–20. Live Excel headers as confirmed during this run: quality_score (17), article_path (18), wp_url (19), asana_subtask_gid (20). There is no wp_post_id column — the post ID has to be embedded in the URL. lib/inventory.md needs to be updated to match.

Quality of the article itself

The subagent's report describes three deliberate calls during generation that are worth flagging for editorial:

  • The Panenka technique is named but not attributed to Antonín Panenka or the 1976 Euro final — kept the article strictly normative, no historical narrative.
  • No conversion-rate statistic ("X% of penalties are converted at elite level") was included — the IFAB rulebook does not provide one and the figure varies meaningfully by competition/season, so an invented number would have failed anti-hallucination.
  • The Bernabéu and Camp Nou references function as stadium-as-atmosphere only — no specific match is referenced, avoiding any hallucinated historical play.

What was tested

Three P1 items from the backlog needed live confirmation before the first real publish (INTEG-01). Rather than wait for the editorial review flow, this run validated all three against a real article without consuming a topic from the inventory.

Steps and findings

ItemTestOutcome
SEO-01 — where does the theme read the meta description from? Fetched a live published post (ID 5608, a real Madrid article in futbol/copa-del-rey) over HTTPS and parsed the rendered head meta tags. Resolved. The theme renders og:description directly from post_excerpt — confirmed exact string match. No <meta name="description"> tag is emitted at all. No SEO plugin in plugins_list (no Yoast, Rank Math, or AIO SEO). Setting post_excerpt = meta_description in posts_create is the right approach; no custom meta keys required.
SEO-02 — does the site emit schema.org JSON-LD without the ig-schema plugin? Parsed every <script type="application/ld+json"> block from the same live post. Resolved. The theme automatically emits two JSON-LD blocks: BreadcrumbList (Home → Category → Article) and NewsArticle (with headline, description from excerpt, datePublished, dateModified, mainEntityOfPage, author, publisher, image, articleSection). Coverage is solid for SEO out of the box. Two nice-to-have refinements were spun off as new backlog items: SEO-03 (add FAQPage for articles with FAQ) and SEO-04 (theme filter to use Article / HowTo instead of NewsArticle for evergreen content).
INFRA-01 — does posts_update accept post_author where posts_create silently ignored it? Created post 16096 with post_author: 77 in posts_create; observed that the response saved post_author: 1 (admin). Then called posts_update with post_author: 77. Resolved. The update response shows before.post_author=1 and after.post_author=77. The workaround is reliable. cmd_new.md step 7.c now does the two-call sequence (create → update with post_author) as part of the production pipeline.

Follow-ups

  • post_author override. Passing post_author: 77 (Pablo's user ID) to posts_create resulted in the saved post having post_author: 1 (admin default). Either the WP adapter ignores the argument on create, or a capability check is missing. Not blocking for the test, but the first real publish should attribute correctly. Two workarounds available: call posts_update immediately after create, or set authorship manually in the admin.
  • Image upload path needs hardening. The featured image had to be downscaled from 1024×1024 PNG (1.3 MB) to a 300×300 JPEG (13 KB) to fit through the interactive Claude Code session. This is a limitation of pasting base64 inline through the assistant's response, not of the MCP. The source.local_path option in media_upload is rejected by the current proxy build (@backend/mcp-proxy 0.2.1), and posting to a public host is correctly blocked as unauthorized exfiltration. The fix is a Python helper that issues the media_upload call directly from the host that already has the image file, returning only the attachment_id back to the pipeline. Tracked as the next item after this milestone.
  • SEO meta fields. The MCP exposes posts_meta_set and a dedicated ig-optimizer adapter, but no meta description was set during this test. The post_excerpt holds the descriptor for now. Before real publish, confirm with the SEO team whether the theme reads meta description from post_excerpt, a Yoast-style meta key, or the ig-optimizer plugin, and wire that into /relevo.
  • Schema.org. The ig-schema plugin is not installed on relevo-com.sub.plus. If structured data matters for SEO (Article, FAQ, BreadcrumbList), either install the plugin or generate JSON-LD in the post body / theme template.
  • Anthropic safety classifier transient outage. During this session the claude-opus-4-7[1m] safety classifier was unavailable for approximately six minutes (14:24 – 14:30 UTC), blocking every MCP and Bash call with "temporarily unavailable, so auto mode cannot determine the safety of <tool>". This is upstream infrastructure, not the MCP. If it recurs, the only recovery is retrying after the classifier returns. Logged here for future operators who see the same symptom.
